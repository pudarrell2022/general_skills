# Auto Commit

## 目标

在不破坏现有改动的前提下，稳定完成一次可追溯的 Git 提交，并向用户汇报结果。

## 执行流程

1. 确认仓库状态
- 运行 `git rev-parse --is-inside-work-tree`。
- 若不是 Git 仓库，先执行 `git init`，然后创建（或补全）`.gitignore`，格式固定为两段：
  - 第一段注释为 `# Folder`，下面填写需要忽略的目录。
  - 第二段注释为 `# File`，下面填写需要忽略的文件。
- `# Folder` 不设默认清单；必须先检查当前目录，只有实际存在且确实需要忽略的目录才可写入（例如存在 `.venv/` 才写 `.venv/`，不存在就不写）。
- `# File` 默认包含 `.DS_Store`。
- 若 `.gitignore` 已存在，优先保留用户已有内容，只补充缺失的段落或条目，不覆盖自定义配置。
- 运行 `git status --short`，若无改动则直接说明“无可提交内容”。

2. 识别提交范围
- 默认提交当前已跟踪与未跟踪的全部改动（`git add -A`）。
- 如果用户明确限定文件或目录，仅暂存指定范围。
- 发现明显无关的大文件、构建产物或敏感文件时，先向用户确认再继续。

3. 生成提交信息
- 先阅读改动摘要：`git diff --stat`，必要时查看 `git diff`。
- 使用中文提交信息，结构为“类型: 简短摘要”。
- 类型优先使用：`feat`、`fix`、`refactor`、`docs`、`test`、`chore`。
- 摘要控制在一行内，聚焦改动结果与意图，不写空泛描述。

4. 执行提交
- 先执行暂存命令，再执行 `git commit -m "<message>"`。
- 若提交失败（如无暂存改动、钩子失败），输出失败原因与下一步建议。

5. 回报结果
- 输出提交哈希短码、提交标题、变更文件数量。
- 可附带最近一次提交信息：`git log -1 --oneline`。

## 质量与安全要求

- 不要改写历史：除非用户明确要求，不执行 `--amend`、`rebase`、`reset --hard`。
- 不要回滚用户未授权的本地改动。
- 遇到冲突状态（如 `MERGE_HEAD` 存在）先提示用户仓库正在合并流程中。
- 提交前后都保持输出简洁，重点给出“做了什么、结果如何、下一步是什么”。

## 快速命令模板

```bash
git status --short
git diff --stat
git add -A
git commit -m "chore: 简要描述本次改动"
git log -1 --oneline
```
