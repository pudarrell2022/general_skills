# Auto Push

## 目标

在保证提交流程规范的前提下，自动完成“提交 + 推送”闭环，并清晰反馈推送结果。

## 执行流程

1. 前置检查
- 运行 `git rev-parse --is-inside-work-tree`。
- 若不是 Git 仓库，先执行 `git init`，然后创建（或补全）`.gitignore`，格式固定为两段：
  - 第一段注释为 `# Folder`，下面填写需要忽略的目录。
  - 第二段注释为 `# File`，下面填写需要忽略的文件。
- `# Folder` 不设默认清单；必须先检查当前目录，只有实际存在且确实需要忽略的目录才可写入（例如存在 `.venv/` 才写 `.venv/`，不存在就不写）。
- `# File` 默认包含 `.DS_Store`。
- 若 `.gitignore` 已存在，优先保留用户已有内容，只补充缺失的段落或条目，不覆盖自定义配置。
- 若存在本次新增的文件或目录，先逐项检查是否属于以下类型：
  - 资源型文件（如图片、CSV 等数据资源）。
  - 敏感信息文件（如密钥、凭据、包含隐私数据的配置/导出文件）。
  - 特大号数据文件（体积明显异常、通常不应入库的二进制或数据集）。
- 对上述类型，若对应忽略规则尚未出现在 `.gitignore`，必须先更新 `.gitignore`，再进入后续提交与推送流程。
- 运行 `git status --short`，了解当前工作区状态。

2. 调用提交技能
- 优先调用 `$auto-commit`，沿用其既有提交规范与安全约束。
- 若 `$auto-commit` 执行失败，则停止流程并返回失败原因，不继续推送。

3. 执行推送
- 在提交成功后，运行 `git push` 推送当前分支到默认远端。
- 若推送失败，输出 Git 原因（如无上游分支、鉴权失败、非快进冲突）并给出下一步建议。

4. 回报结果
- 输出最新提交信息：`git log -1 --oneline`。
- 输出推送结果摘要：目标远端、分支名、是否成功。

## 质量与安全要求

- 不改写历史：除非用户明确要求，不执行 `--force`、`push --force-with-lease`、`rebase`、`reset --hard`。
- 不绕过 `$auto-commit` 的安全检查与提交流程。
- 遇到冲突或权限问题时，先报告现状与可选处理方式，再等待用户决定。
- 全程保持简洁输出，重点说明“做了什么、结果如何、下一步是什么”。

## 快速命令模板

```bash
# 1) 先调用提交技能
$auto-commit

# 2) 再推送
git push

# 3) 查看最新提交
git log -1 --oneline
```
